# Промпт для реализации task-state workflow в чате

Ты senior fullstack engineer и system analyst. Реализуй в текущем приложении механизм «задача в чате» с формализованным конечным автоматом состояний. Пиши production-ready код, но без избыточной сложности.

## 1) Цель
Добавить в чат режим выполнения выбранной задачи, где диалог идет только по алгоритму задачи и по допустимым переходам состояний:
- `planning -> execution`
- `execution -> validation`
- `validation -> execution`
- `validation -> done`

Любые другие переходы запрещены.

## 2) Новая настройка в UI
В блоке настроек чата (в самый низ) добавить `select` «Задача»:
- `нет задачи` (значение по умолчанию)
- `сделать промпт для создания фронтенд приложения`

Требования:
- Выбрать задачу можно только при создании нового чата.
- После старта чата значение задачи фиксируется и не меняется.
- Если выбрано `нет задачи`, task-state алгоритм не применяется, чат работает как сейчас.
- Архитектура должна быть расширяемой для добавления новых задач (registry/enum/конфиг задач).

## 3) Модель состояния задачи
Для чата с выбранной задачей ввести и сохранять объект состояния:
- `taskId`
- `stage` (`planning | execution | validation | done`)
- `currentStep` (человекочитаемое описание текущего шага)
- `expectedAction` (что ждем от пользователя сейчас)
- `planningQuestions` (список вопросов)
- `planningAnswers` (карта ответов по каждому вопросу)
- `lastGeneratedPrompt` (последняя версия итогового промпта)
- `validationResult` (замечания/рекомендации ревью)
- служебные поля (`createdAt`, `updatedAt`, `version`)

Требования к хранению:
- Состояние должно переживать перезапуск страницы/приложения.
- Использовать существующую стратегию persistence проекта (встроиться в текущие storage/memory сущности, не ломая обратную совместимость).

## 4) Строгий формат ответов ассистента в task-mode
Для любого ответа в task-mode ассистент обязан выводить 3 блока в начале:
1. `Этап задачи: <planning|execution|validation|done>`
2. `Текущий шаг: <текст>`
3. `Ожидаемое действие: <текст>`

Далее — содержательная часть ответа.

## 5) Логика задачи "сделать промпт для создания фронтенд приложения"

### Stage: planning
Цель: собрать вводные через 5–10 вопросов по стеку и подходам.

Правила:
- На старте planning сгенерировать 5–10 конкретных вопросов.
- Пользователь может отвечать частями в нескольких сообщениях.
- Не переходить дальше, пока не заполнены ответы на ВСЕ вопросы.
- Если пользователь отвлекается от темы или просит перейти дальше раньше времени:
  - вежливо напомнить, что нужны ответы на все вопросы для продолжения,
  - перечислить, по каким вопросам еще нет ответа.
- Пока есть незаполненные ответы, оставаться в `planning`.

Условие перехода:
- Когда все вопросы заполнены -> `planning -> execution`.

### Stage: execution
Цель: сформировать качественный промпт «как senior аналитик / frontend разработчик» для передачи code assistant.

Правила:
- Использовать все ответы planning.
- Сформировать структурированный итоговый промпт (цель, контекст, ограничения, стек, нефункциональные требования, критерии приемки, формат результата).
- Сохранить результат в `lastGeneratedPrompt`.

После выдачи промпта пользователю:
- автоматически перейти в `validation`.

### Stage: validation
Цель: провалидировать сгенерированный промпт через тот же LLM (внутренний ревью-прогон).

Правила:
- Запустить review последней версии промпта с ролью «senior reviewer».
- Если есть замечания/рекомендации:
  - показать их пользователю,
  - задать уточняющие вопросы для правок,
  - ждать ответы пользователя,
  - после получения ответов перейти `validation -> execution` для пересборки промпта.
- Если пользователь подтверждает, что результат устраивает:
  - перейти `validation -> done`,
  - вывести финальный промпт.

### Stage: done
Цель: завершение сценария.

Правила:
- Сообщить, что задача завершена.
- Финальный промпт показать в последнем ответе.
- При попытке продолжить task-flow после done — сообщить, что сценарий завершен и предложить создать новый чат для новой задачи.

## 6) Ограничения поведения (guardrails)
- В task-mode нельзя «перепрыгивать» этапы.
- Нельзя выполнять недопустимые переходы FSM.
- Нельзя терять уже собранные ответы planning.
- Любое пользовательское сообщение обрабатывается в рамках текущего этапа.
- Если пользователь просит «отменить задачу» в середине, не делать ранний выход: сценарий доводится до `done`.

## 7) Архитектурные требования
- Встроить решение в существующие сущности/хуки/хранилища чата.
- Выделить отдельный слой orchestration для task-state логики (чтобы добавлять новые задачи без переписывания ядра чата).
- Добавить типы и контракты для:
  - конфигурации задачи,
  - состояния задачи,
  - обработчика переходов,
  - валидатора допустимости перехода.
- Реализовать таблицу переходов (transition map) как single source of truth.

## 8) Наблюдаемость и диагностика
Добавить безопасный debug-лог (без чувствительных данных):
- текущий stage,
- входное событие,
- примененный переход,
- причина отказа перехода (если есть).

## 9) Тесты (обязательно)
Покрыть тестами минимум:
- корректные переходы FSM,
- запрет недопустимых переходов,
- зависание в planning до заполнения всех ответов,
- переход execution -> validation,
- циклы validation -> execution при замечаниях,
- переход validation -> done при подтверждении пользователя,
- persistence/восстановление состояния после перезапуска,
- поведение при `нет задачи` (старый флоу без изменений).

## 10) Критерии приемки
Считаем задачу выполненной, если:
- В UI есть select задачи в настройках (внизу), с нужными значениями.
- Выбор задачи доступен только при создании нового чата.
- Для выбранной задачи работает строгий FSM только с разрешенными переходами.
- Ассистент в task-mode всегда показывает: `Этап задачи`, `Текущий шаг`, `Ожидаемое действие`.
- Planning не завершается без ответов на все вопросы.
- Validation выполняется внутренним ревью тем же LLM.
- Состояние сохраняется и корректно восстанавливается.
- `нет задачи` не меняет текущее поведение системы.
