# Добавление планировщика (scheduler)

## Логика работы по шагам

1. Пользователь отправляет сообщение.
2. В основной LLM-контекст добавляются system-инструкции планировщика (`Organizer Scheduler Mode`).
3. Если интент про планирование действия, LLM возвращает JSON:

```json
{
  "type": "organizer",
  "method": "scheduler",
  "value": "plan_action",
  "setting": {}
}
```

4. Runtime парсит JSON `organizer/scheduler/plan_action`.
5. Проверяется общий MCP-флаг:
- если MCP выключен: в чат отправляется `MCP не включен, сначала включите MCP и повторите запрос`;
- если MCP включен: запускается wizard планировщика.

6. Wizard в этом же чате показывает вводное сообщение и шаги сбора данных:
- `repeat`: сколько раз (`1..100` или `всегда`);
- `interval`: интервал в секундах (`10..86400`);
- `action`: текст действия для выполнения.

7. На каждом шаге идет валидация:
- при ошибке запрашивается ввод повторно с уточнением;
- `отмена` на любом шаге завершает wizard и возвращает чат в обычный режим.

8. После валидного ввода создается событие:
- `id`, `action`, `intervalSeconds`, `repeat`, `createdAt`, `nextRunAt`;
- сохраняется в `localStorage` (`scheduled_events_v1`);
- в чат отправляется подтверждение создания.

9. Планировщик запускает таймер на ближайшее событие и выполняет события через **глобальную очередь**.

10. Если в момент срабатывания идет обычный запрос (`status=loading`), событие ждет.

11. При срабатывании события:
- выполняется LLM-запрос с `action` пользователя;
- в контекст включаются `profile` + `MCP` + `organizer` system-инструкции;
- если приходит JSON-команда (например `mcp` или `organizer`), она исполняется штатной логикой;
- в чат пишется только итоговый текст (сырые JSON не показываются).

12. После выполнения:
- для `count` уменьшается `remainingRuns`, при `0` событие удаляется;
- для `always` событие остается;
- `nextRunAt` пересчитывается как `now + interval`;
- пропущенные интервалы не догоняются, планируется следующий запуск.

13. В UI слева под чатами отображается сворачиваемый список событий:
- title = `action + повторения + createdAt (dd.MM.yyyy HH:mm)`;
- удаление события через подтверждение `Удалить запланированное действие?`.

14. При ошибке выполнения события в чат пишется:
- `Планировщик "<action>" завершился с ошибкой: ...`.

## Итоговый prompt (Organizer Scheduler Mode)

```md
# Organizer Scheduler Mode

mcp_enabled = {{true|false}}

Применяй эти правила только к основному ответу ассистента.

Если запрос пользователя однозначно про планирование действия (напоминание/запланировать задачу/активность/вызвать планировщик), верни только JSON без markdown и пояснений:
{
  "type": "organizer",
  "method": "scheduler",
  "value": "plan_action",
  "setting": {}
}

Возвращай этот JSON при явном интенте запланировать действие. Примеры интента:
- "планировщик"
- "запланируй задачу"
- "запланируй активность"
- "вызови планировщик"
- любые аналогичные просьбы запланировать действие.

Если интент неоднозначный или не про планирование действия:
- не возвращай JSON organizer;
- отвечай обычным текстом.

Важно:
- не добавляй дополнительные поля;
- не добавляй текст до или после JSON.
```
