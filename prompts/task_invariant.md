# Промпт для fixed questions и invariant-check в задаче frontend_app_prompt

Ты senior frontend engineer и system analyst. В текущем приложении нужно доработать существующую задачу `frontend_app_prompt` в 2 этапа: сначала зафиксировать planning-вопросы на стороне приложения, затем добавить механизм инвариантов для этой же задачи. Пиши production-ready код без лишней сложности, с сохранением текущей архитектуры и обратной совместимости.

## Общие ограничения

- Работай только в рамках текущего приложения и существующей архитектуры task workflow.
- Используй текущую задачу `frontend_app_prompt`, не создавай новую.
- Не ломай текущий task-mode, а расширяй его.
- Если задача не выбрана, поведение приложения не меняется.
- Решение должно быть расширяемым для будущих задач с собственными fixed questions и invariants.
- Используй существующий persistence-подход проекта, чтобы новые поля корректно сохранялись и восстанавливались.

## Этап 1. Fixed questions для задачи `сделать промпт для создания фронтенд приложения`

### Цель

Убрать вариативность planning-вопросов со стороны LLM и зафиксировать их на стороне frontend-приложения в константе задачи.

### Обязательный список вопросов

Задача `frontend_app_prompt` должна использовать строго этот список:

1. Какой тип фронтенд приложения создаем (лендинг, SaaS, админка, маркетплейс и т.д.)?
2. Какой основной стек обязателен (React/Vue/Svelte, TypeScript, роутинг, состояние)?
3. Какая дизайн-система (MUI, Ant, Tailwind)?
4. Какие требования к API-интеграции (REST, auth, обработка ошибок, retries)?
5. Какие нефункциональные требования важны (производительность, accessibility, SEO, i18n)?
6. Какие требования к качеству кода и тестам (unit, integration, e2e, coverage, lint/format)?
7. Нужны ли CI/CD, окружения и стратегия деплоя (preview/stage/prod)?
8. Какие ограничения и критерии приемки у результата (сроки, команды, готовые модули, Definition of Done)?

### Требования этапа 1

- Planning должен работать только по фиксированным вопросам из константы.
- LLM не должен генерировать planning-вопросы динамически.
- Пользователь может отвечать частями в нескольких сообщениях.
- Переход дальше возможен только когда собраны ответы на все 8 вопросов.
- Формулировка самой задачи и ее workflow должны явно использовать предустановленные пункты.
- Текущая state machine должна сохраниться и продолжить работать предсказуемо.

## Этап 2. Инварианты для задачи

### Цель

Добавить сущность инвариантов для выбранной задачи. Инвариант ограничивает допустимые ответы пользователя и итоговый результат по задаче. Если инвариант нарушен, workflow не должен двигаться дальше, пока пользователь не исправит ответ.

## UI-изменения

- Если выбрана не пустая задача, под блоком задачи показывать блок инвариантов для этой задачи.
- Если задача не выбрана, под блоком задачи ничего не отображать.
- Если у задачи нет инвариантов, блок инвариантов не отображать.
- Для задачи с инвариантами добавить чекбокс включения/отключения инвариантов.
- Если чекбокс включен, инварианты активны и обязательны.
- Если чекбокс выключен, инварианты не применяются и блокирующая логика не работает.
- Блок инвариантов только отображает список инвариантов задачи в read-only виде.
- В блоке инвариантов не нужны статусы выполнения, история ошибок или прогресс-подсветка.

## Инварианты для задачи `frontend_app_prompt`

### Вопрос 2

Вопрос:
`Какой основной стек обязателен (React/Vue/Svelte, TypeScript, роутинг, состояние)?`

Инвариант:

- допускается только один из трех фреймворков: `React`, `Vue`, `Svelte`
- никакой другой фреймворк не принимается
- отсутствие фреймворка не допускается

### Вопрос 3

Вопрос:
`Какая дизайн-система (MUI, Ant, Tailwind)?`

Инвариант:

- допускается только одна из перечисленных дизайн-систем: `MUI`, `Ant`, `Tailwind`
- никакая другая дизайн-система не принимается
- вариант без дизайн-системы не допускается

### Вопрос 4

Вопрос:
`Какие требования к API-интеграции (REST, auth, обработка ошибок, retries)?`

Инвариант:

- допускается только `REST`
- `GraphQL` не допускается
- любая другая система для API не допускается

## Логика применения инвариантов

- Инварианты применяются только если включены чекбоксом.
- Проверка инвариантов должна выполняться отдельным LLM-запросом.
- Проверка должна выполняться два раза:
  1. при обработке пользовательских ответов в `planning`
  2. повторно при дальнейшей генерации и проверке результата, чтобы итог по задаче тоже не нарушал инварианты
- Проверка должна быть жесткой и блокирующей.
- Если пользовательский ответ не проходит инвариант, выполнение задачи не продолжается.
- Если итоговый результат не проходит инвариант, выполнение задачи тоже не продолжается.
- Пока пользователь не даст корректный ответ-сообщение, workflow не должен двигаться дальше.
- При нарушении инварианта нельзя переходить к следующему этапу.
- При успешной инвариантной проверке выполнение продолжается по обычному task workflow.

## Строгий формат ответа при нарушении инварианта

Если сработал инвариант и найдено противоречие, ассистент обязан отвечать в строгом формате:

1. первая строка: `**Этап задачи:** <planning|execution|validation>`
2. вторая строка: `**Текущий шаг:** <текст>`
3. третья строка: `**Ожидаемое действие:** <текст>`
4. затем пустая строка
5. затем текст строго по шаблону:

`**Нарушение инварианта:**`
`- Инвариант: <текст инварианта>.`
`- Вопрос: <номер и текст вопроса>.`
`- Что не так: ответ противоречит ограничению задачи.`
`- Что сделать: исправьте ответ по инварианту.`

Сделай этот формат единым и обязательным для всех invariant violations.

## Архитектурные требования

- Встроить описание инвариантов в существующую модель task configuration.
- Вынести fixed questions и invariants в конфиг/константы задачи.
- Не хардкодить бизнес-правила в случайных местах workflow.
- Добавить типы и контракты для:
  - fixed questions задачи
  - описания инварианта
  - набора инвариантов задачи
  - флага включения/отключения инвариантов
  - результата инвариантной проверки
- Выделить отдельный orchestration/validation слой для invariant-check.
- Избегать лишнего дублирования правил между UI, workflow и validation.
- Корректно сохранять и восстанавливать состояние флага инвариантов и связанных данных задачи.

## Требования к workflow

- Этап 1 и этап 2 должны быть реализованы последовательно в рамках одной задачи.
- После внедрения fixed questions planning должен использовать только предустановленные пункты.
- При активных инвариантах planning не может завершиться, если ответы на вопросы 2, 3, 4 нарушают ограничения.
- После генерации результата должна выполняться повторная инвариантная проверка.
- Если итоговый результат нарушает инварианты, задача не должна завершаться успешно.
- Пользователю должно быть явно сказано, какой инвариант нарушен и что нужно исправить.
- Пока пользователь не исправит нарушение, workflow не продвигается.

## Тесты

Добавь тесты минимум на следующее:

- fixed questions действительно используются вместо любой динамической генерации вопросов
- planning не завершается без ответов на все 8 фиксированных вопросов
- UI показывает блок инвариантов только если выбрана задача с инвариантами
- UI не показывает блок инвариантов, если задача не выбрана или если инвариантов нет
- чекбокс корректно включает и отключает инварианты
- при выключенных инвариантах задача идет по обычному flow
- при включенных инвариантах нарушение ответа на вопрос 2 блокирует продвижение
- при включенных инвариантах нарушение ответа на вопрос 3 блокирует продвижение
- при включенных инвариантах нарушение ответа на вопрос 4 блокирует продвижение
- сообщение о нарушении инварианта выводится в строгом формате
- повторная инвариантная проверка итогового результата тоже блокирует workflow при нарушении
- после корректировки ответа пользователем workflow может продолжиться
- persistence корректно сохраняет состояние флага инвариантов и связанные данные задачи

## Критерии приемки

Считать задачу выполненной, если:

- задача `frontend_app_prompt` использует только фиксированные вопросы из константы
- planning больше не зависит от вариативной генерации вопросов
- для задачи можно описать и отобразить инварианты
- блок инвариантов показывается только в нужных случаях
- инварианты можно включать и отключать чекбоксом
- при включенных инвариантах проверка идет отдельным LLM-запросом
- проверка выполняется дважды: на пользовательских ответах и на дальнейшем результате
- при нарушении инварианта workflow блокируется до исправления
- пользователю выводится строгий текст о нарушении инварианта
- при выключенных инвариантах текущее поведение не ломается
