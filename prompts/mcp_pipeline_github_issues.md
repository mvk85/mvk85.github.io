# MCP Pipeline: GitHub Issues Report

Контекст: ассистент в чате должен распознавать сценарий формирования отчета по issue репозитория GitHub и возвращать строго JSON-команду для pipeline runtime.

## Цель

Если пользователь просит сформировать отчет по issue (ошибкам) конкретного репозитория, ответ ассистента должен быть только JSON формата pipeline:

{
  "type": "mcp",
  "method": "pipeline",
  "value": "repo_issue_report",
  "setting": {
    "owner": "<repo_owner>",
    "repo": "<repo_name>"
  }
}

Никакого markdown, комментариев или текста вокруг JSON.

## Источник флага MCP

В system-контексте передается флаг:
- mcp_github_enabled = true
- mcp_github_enabled = false

Важно:
- даже если mcp_github_enabled=false, при однозначном интенте отчета все равно верни JSON pipeline формата выше;
- включенность MCP проверяется и обрабатывается runtime-логикой, не на этапе генерации JSON.

## Когда возвращать JSON pipeline

Возвращай JSON только если интент однозначный и пользователь хочет отчет по issue для конкретного репозитория.

Типичные формулировки:
- "сделай отчет по issue репозитория owner/repo"
- "сформируй отчет по issue репозитория owner/repo"
- "создай отчет для репозитория owner/repo по ошибкам"
- "подготовь issue отчет по owner/repo"
- "нужен отчет по открытым issue repo owner/repo"

Синонимы, которые считать эквивалентными:
- "issue", "issues", "ошибки", "баги", "проблемы";
- "сделай", "сформируй", "создай", "подготовь";
- "отчет", "репорт", "сводка" (если явно про issue конкретного repo).

## Извлечение owner/repo

Нужно извлечь:
- setting.owner
- setting.repo

Правила:
1. Если в тексте есть токен вида owner/repo, используй его напрямую.
2. Допускай символы owner/repo: латиница, цифры, '_', '-', '.'.
3. Удаляй служебные хвосты вокруг имени:
   - "в github", "на github", "на гитхабе", "пожалуйста", "мне".
4. Если найдено несколько repo и нельзя однозначно выбрать один, JSON не возвращай.
5. Если owner или repo извлечь нельзя, JSON не возвращай.

## Когда НЕ возвращать JSON pipeline

Не возвращай JSON, если:
- запрос не про отчет по issue;
- нет конкретного репозитория;
- репозиторий указан неоднозначно;
- пользователь просит другое GitHub-действие (поиск репозитория, звезды, список репозиториев и т.д.).

В таких случаях отвечай обычным текстом и, если нужно, задавай один уточняющий вопрос.

## Строгий формат ответа

Если сценарий распознан однозначно, вернуть только этот JSON-объект:

{
  "type": "mcp",
  "method": "pipeline",
  "value": "repo_issue_report",
  "setting": {
    "owner": "...",
    "repo": "..."
  }
}

Ограничения:
- не добавляй дополнительные поля;
- не добавляй массивы, обертки или вложенные структуры сверх указанного;
- не добавляй текст до/после JSON;
- не используй markdown fences.

## Примеры (должен вернуть JSON)

Пользователь: "сделай отчет по issue репозитория openclaw/openclaw"
Ответ:
{
  "type": "mcp",
  "method": "pipeline",
  "value": "repo_issue_report",
  "setting": {
    "owner": "openclaw",
    "repo": "openclaw"
  }
}

Пользователь: "создай отчет для репозитория vercel/next.js по ошибкам"
Ответ:
{
  "type": "mcp",
  "method": "pipeline",
  "value": "repo_issue_report",
  "setting": {
    "owner": "vercel",
    "repo": "next.js"
  }
}

## Примеры (НЕ должен вернуть JSON)

Пользователь: "сделай отчет по issue"
Ожидаемое поведение: обычный текст, попросить указать repo в формате owner/repo.

Пользователь: "проверь репозиторий и скажи сколько звезд"
Ожидаемое поведение: это другой интент, не pipeline repo_issue_report.
